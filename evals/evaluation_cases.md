# Evaluation Cases

このファイルは、このスキルの評価用ケース集です。`examples.md` とは分離し、同じ案件を使い回さないでください。

各 case では、次を確認してください。

- 必須で聞くべき項目
- 進んではいけない条件
- 期待される最終出力種別

## Case 1: 最小入力の物流案件

入力:

```text
出荷ラベル用途で自社のラベルプリンターを提案したいです。
次回訪問で使える提案資料を作ってください。
```

必須で聞く項目:

- 読者
- 既存運用
- 提案範囲
- 導入対象規模

進んではいけない条件:

- 読者が不明
- 既存運用が不明
- 提案範囲が不明

期待される最終出力種別:

- `追加確認`

## Case 2: 中程度の飲食案件

入力:

```text
飲食チェーン向けに食材ラベル用途で自社のラベルプリンターを提案したいです。
店舗でラベル作成の手間があり、記載ミスも減らしたいです。
本体、消耗品、初期設定支援まで含めて考えています。
価格と納期はまだ未確定です。
```

必須で聞く項目:

- 読者
- 既存運用
- 効果を語る重視項目

進んではいけない条件:

- 読者が不明
- 既存運用が不明
- 提案範囲が不明

期待される最終出力種別:

- 読者、既存運用、提案範囲が埋まるまでは `追加確認`
- それらが埋まり、規模や効果指標が弱い場合は `仮説前提のたたき台`

## Case 3: 情報十分な製造案件

入力:

```text
製造業の工場向けに、工程ラベル用途で自社のラベルプリンターを提案したいです。
現場で手書きラベルが残っていて、記載ミスと貼り直しが課題です。
工場長と生産管理部向けの資料にしたいです。
本体、消耗品、初期設定支援を提案対象に含めます。
まずは2ライン、計4台から始めたいです。
```

必須で聞く項目:

- 原則なし
- 必要なら効果を語る指標だけ確認

進んではいけない条件:

- 顧客固有の課題が前半に出ない
- 提案対象が具体化されない
- 読者に合う説得順序にならない

期待される最終出力種別:

- `完成した提案資料`

## Case 4: 情報不足の小売案件

入力:

```text
小売向けにラベルプリンターの提案資料を作りたいです。
店舗運用を改善したいです。
```

必須で聞く項目:

- 読者
- 既存運用
- 提案範囲
- 用途

進んではいけない条件:

- 読者が不明
- 既存運用が不明
- 提案範囲が不明

期待される最終出力種別:

- `追加確認`

## Case 5: 比較が必要な案件

入力:

```text
物流会社向けに、自社のラベルプリンターを既存他社機から置き換える提案をしたいです。
現場では再印字とラベル貼付ミスが課題です。
本部の物流部長向けに提案したいです。
```

必須で聞く項目:

- 比較対象
- 既存運用
- 提案範囲

進んではいけない条件:

- 比較対象が不明
- 既存運用が不明

期待される最終出力種別:

- 比較対象または既存運用が不明な間は `追加確認`
- 両方埋まり、他の必須確認項目も満たせば `完成した提案資料` または `仮説前提のたたき台`

## 共通合格条件

- 各 case で期待される最終出力種別が一意に判定できる
- 必須確認項目が欠けたまま `Compose` や `Render` に進まない
- `追加確認` の case では PPTX を生成しない
- `仮説前提のたたき台` の case では、必須チャット文言が出る
- `完成した提案資料` の case では、顧客固有の課題が前半に反映されている
