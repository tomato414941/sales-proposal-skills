# Evaluation Cases

このファイルは、このスキルの評価用ケース集です。`examples.md` とは分離し、同じ案件を使い回さないでください。

各 case では、次を確認してください。

- `Frame` で整理される骨格
- 必須で聞くべき項目
- 選択肢を返すべき論点
- 進んではいけない条件
- 期待される最終出力種別

## Case 1: 最小入力の物流案件

入力:

```text
出荷ラベル用途で自社のラベルプリンターを提案したいです。
次回訪問で使える提案資料を作ってください。
```

`Frame` で整理される骨格:

- 顧客: 物流会社
- 用途: 出荷ラベル
- 目的: 次回訪問用

必須で聞く項目:

- 読者
- 既存運用
- 提案範囲
- 営業フェーズ
- 導入対象規模
- 自社商材の押しどころ

選択肢を返すべき論点:

- 現場責任者向けに寄せるか、物流部長向けに寄せるか、両方にするか
- 初回訪問の叩き台にするか、正式提案にするか

進んではいけない条件:

- 読者が不明
- 既存運用が不明
- 提案範囲が不明
- 営業フェーズが不明

期待される最終出力種別:

- `追加確認`

質問手段の期待:

- 質問 UI / 質問ツールが使える場合は、必須項目をまとめて確認する
- 質問 UI が使えない場合は、通常の会話で同じ確認を返す
- 読者の方向性が割れる場合は、短い選択肢で決めてもらう

## Case 2: 中程度の飲食案件

入力:

```text
飲食チェーン向けに食材ラベル用途で自社のラベルプリンターを提案したいです。
店舗でラベル作成の手間があり、記載ミスも減らしたいです。
本体、消耗品、初期設定支援まで含めて考えています。
価格と納期はまだ未確定です。
```

`Frame` で整理される骨格:

- 顧客: 飲食チェーン
- 用途: 食材ラベル
- 提案対象: 本体、消耗品、初期設定支援
- 制約: 価格と納期は未確定

必須で聞く項目:

- 読者
- 既存運用
- 営業フェーズ
- 効果を語る重視項目

選択肢を返すべき論点:

- 本部向けに標準化を前に出すか、店舗向けに運用負荷改善を前に出すか
- 初回ヒアリング用にするか、正式提案にするか

進んではいけない条件:

- 読者が不明
- 既存運用が不明
- 提案範囲が不明
- 営業フェーズが不明

期待される最終出力種別:

- 読者、既存運用、提案範囲が埋まるまでは `追加確認`
- 初回訪問なら、規模や効果指標が弱い場合は `仮説前提のたたき台`
- 正式提案なら、規模や効果指標が弱い場合でも確認に戻る

質問手段の期待:

- 質問 UI / 質問ツールが使える場合は、必須確認項目を一括で確認する
- 質問 UI が使えない場合は、通常の会話で同じ確認を返す
- 資料の方向性が割れる場合は、選択肢を返して決めてもらう

## Case 3: 情報十分な製造案件

入力:

```text
製造業の工場向けに、工程ラベル用途で自社のラベルプリンターを提案したいです。
現場で手書きラベルが残っていて、記載ミスと貼り直しが課題です。
工場長と生産管理部向けの資料にしたいです。
本体、消耗品、初期設定支援を提案対象に含めます。
まずは2ライン、計4台から始めたいです。
```

`Frame` で整理される骨格:

- 顧客: 製造業の工場
- 用途: 工程ラベル
- 読者候補: 工場長、生産管理部
- 提案対象: 本体、消耗品、初期設定支援
- 営業フェーズ候補: 初回訪問または正式提案

必須で聞く項目:

- 営業フェーズが曖昧なら確認
- 必要なら効果を語る指標だけ確認

選択肢を返すべき論点:

- summary 先行で見せるか、現場課題先行で見せるか
- 初回訪問向けに軽くまとめるか、正式提案として導入イメージまで厚くするか

進んではいけない条件:

- 顧客固有の課題が前半に出ない
- 提案対象が具体化されない
- 読者に合う説得順序にならない
- 営業フェーズ未確定のまま資料の重さを決め打ちする

期待される最終出力種別:

- `完成した提案資料`

質問手段の期待:

- 質問 UI / 質問ツールは不要か、必要でも最小限
- 構成の重心だけ迷う場合は、短い選択肢で決めてもらう

## Case 4: 情報不足の小売案件

入力:

```text
小売向けにラベルプリンターの提案資料を作りたいです。
店舗運用を改善したいです。
```

`Frame` で整理される骨格:

- 顧客: 小売
- 用途: 店舗運用改善
- 提案対象: ラベルプリンター

必須で聞く項目:

- 読者
- 既存運用
- 提案範囲
- 営業フェーズ
- 用途

選択肢を返すべき論点:

- 値引きラベル、棚ラベル、バックヤード用途のどれを軸にするか
- 初回訪問の叩き台にするか、正式提案にするか

進んではいけない条件:

- 読者が不明
- 既存運用が不明
- 提案範囲が不明
- 営業フェーズが不明

期待される最終出力種別:

- `追加確認`

質問手段の期待:

- 質問 UI / 質問ツールが使える場合は、確認に入る
- 質問 UI が使えない場合は、通常の会話で確認に入る
- 用途が複数ありえる場合は、選択肢で方向性を返す

## Case 5: 比較が必要な案件

入力:

```text
物流会社向けに、自社のラベルプリンターを既存他社機から置き換える提案をしたいです。
現場では再印字とラベル貼付ミスが課題です。
本部の物流部長向けに提案したいです。
```

`Frame` で整理される骨格:

- 顧客: 物流会社
- 用途: ラベル貼付運用改善
- 読者: 本部の物流部長
- 方向性候補: 比較提案 or 課題整理提案

必須で聞く項目:

- 比較対象
- 既存運用
- 提案範囲
- 営業フェーズ

選択肢を返すべき論点:

- 比較提案として組み立てるか、課題整理提案として組み立てるか
- 比較検討フェーズの資料にするか、正式提案として比較を補足に留めるか

進んではいけない条件:

- 比較対象が不明
- 既存運用が不明
- 営業フェーズが不明

期待される最終出力種別:

- 比較対象または既存運用が不明な間は `追加確認`
- 初回訪問なら、両方埋まり他の必須確認項目も満たせば `仮説前提のたたき台` 可
- 比較検討または正式提案なら、両方埋まり他の必須確認項目も満たしたうえで `完成した提案資料`

質問手段の期待:

- 質問 UI / 質問ツールが使える場合は、比較対象と既存運用を優先して確認する
- 質問 UI が使えない場合は、通常の会話で同じ確認を返す
- 比較の立て方自体が割れる場合は、選択肢で決めてもらう

## 共通合格条件

- 各 case で期待される最終出力種別が一意に判定できる
- 必須確認項目が欠けたまま `Compose` や `Render` に進まない
- 必須確認項目が欠けている case では、質問 UI / 質問ツール優先の確認フェーズに入る
- 質問 UI がない環境でも、通常の会話で同じ確認内容を返せる
- 方向性が割れる case では、選択肢を返すべき論点を勝手に一つへ決め打ちしない
- `追加確認` の case では PPTX を生成しない
- `仮説前提のたたき台` の case では、必須チャット文言が出る
- `完成した提案資料` の case では、顧客固有の課題が前半に反映されている
- 営業フェーズが曖昧なまま資料の重さを決めない
- 自社商材を選ぶ理由が読めない資料を通さない

## Case 6: 同一案件の reader 差分

入力:

```text
物流会社向けに、出荷ラベル用途で自社のラベルプリンターを提案したいです。
手書き運用による再発行と貼付ミスが課題です。
```

期待:

- 本部向けと現場向けが自然に分かれる場合は、読者を選択肢化する
- 本部向けを選んだ場合は、標準化、展開性、投資判断が前に出る
- 現場向けを選んだ場合は、作業負荷、ミス低減、使いやすさが前に出る

## Case 7: 同一案件の営業フェーズ差分

入力:

```text
飲食チェーン向けに、食材ラベル用途で自社のラベルプリンターを提案したいです。
店舗ごとの運用ばらつきと記載ミスが気になっています。
```

期待:

- 初回訪問 / 比較提案 / 見積前提 のどれで進めるか曖昧なら選択肢化する
- 初回訪問なら、課題整理と確認事項が前に出る
- 比較提案なら、比較対象と判断論点が前に出る
- 見積前提なら、提案範囲と導入イメージが前に出る

## Case 8: 同一案件の初回訪問と正式提案差分

入力:

```text
外食チェーン向けに、食材ラベル用途で自社のラベルプリンターを提案したいです。
店舗の手書き運用と記載ミスが課題だと聞いています。
本体、消耗品、初期設定支援を提案したいです。
```

期待:

- 初回訪問なら、仮説前提のたたき台として進めてよい
- 正式提案なら、規模、効果指標、導入条件が弱いまま完成資料にしない
- 同じ案件でも、初回訪問と正式提案で資料の重さと次アクションが変わる

## Case 9: 課題はあるが why us が弱い案件

入力:

```text
物流会社向けに、出荷ラベル用途で自社のラベルプリンターを提案したいです。
再発行と貼付ミスが課題です。
本部向けの提案資料にしたいです。
```

期待:

- 課題と読者だけで資料を閉じず、自社商材として何を強みとして押すかを確認する
- `なぜ自社商材か` が弱いまま完成資料にしない
- 最終資料では、課題、提案、why us、次アクションが読める
