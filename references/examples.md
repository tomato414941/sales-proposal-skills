# Examples

このファイルは、営業担当者が実際に入力しそうな自然文と、それに対してこのスキルがどう質問し、どう選択肢を返し、どう資料化するかを見るための説明用サンプルです。

評価にはこのファイルの案件をそのまま使わず、[evaluation_cases.md](../evals/evaluation_cases.md) を使ってください。

## 例1: 物流向けの最小入力

入力:

```text
地方の物流会社向けに、出荷ラベル用途で自社のラベルプリンターを提案したいです。
次回訪問で使える提案資料を作ってください。
```

最初に整理する骨格:

- 顧客: 地方の物流会社
- 商材: ラベルプリンター
- 用途: 出荷ラベル
- 目的: 次回訪問用の提案資料

返す追加質問の例:

- 現在の出荷ラベル運用は手書きですか、既存プリンターがありますか
- 誰に見せる資料ですか。現場責任者向けか、物流部長向けか、両方ですか
- 本体だけでなく、消耗品や導入支援も含めて提案しますか
- 最初は何拠点、何台程度を想定していますか

その後の流れ:

- 利用可能なら質問 UI / 質問ツールで上の確認をまとめて聞く
- 質問 UI が使えない場合は、通常の会話で同じ内容をまとめて返す
- 読者が複数候補なら、`現場責任者向けに寄せる / 物流部長向けに寄せる / 両方に配慮する` のように短い選択肢で返して決めてもらう
- 営業フェーズが曖昧なら、`初回訪問の叩き台 / 比較提案 / 正式提案` のように確認する
- 必要なら骨格を artifact で見せて方向確認する
- 回答を反映して提案の勝ち筋を決める
- 自社商材ならではの押しどころを 1 つは明確にする
- 現場向けなら作業負荷とミス低減、本部向けなら標準化と展開性を前に出す
- スライド構成では、前半で課題と提案価値、最後で次アクションを明確にする
- 持参可能な PPTX を返す

## 例2: 飲食チェーン向けの中程度入力

入力:

```text
飲食チェーン向けに、食材ラベル用途で自社のラベルプリンターを提案したいです。
店舗でラベル作成の手間がかかっていて、手書き運用による記載ミスもありそうです。
本体、消耗品、初期設定支援まで含めて考えています。
価格や納期はまだ未確定なので断定しないでください。
```

最初に整理する骨格:

- 顧客: 飲食チェーン
- 商材: 本体、消耗品、初期設定支援
- 課題: ラベル作成の手間、記載ミス
- 制約: 価格と納期は未確定

返す追加質問の例:

- 誰に見せる資料ですか
- 現在のラベル運用は手書き中心ですか、それとも既存プリンターがありますか
- 最初は何店舗、何台程度を想定していますか
- 食品安全、オペレーション効率、標準化のうち、何を最も重視していますか

選択肢で返す例:

- `本部向けに標準化と展開性を前に出す / 店舗向けに運用負荷とミス低減を前に出す / 両方に配慮した中間案にする`
- `初回ヒアリング用の課題整理提案にする / 正式提案として具体導入を前提にする`

その後の流れ:

- 利用可能なら質問 UI / 質問ツールで確認をまとめて聞く
- 読者と既存運用を確定してからスライド構成に入る
- 初回ヒアリング用か、正式提案かで資料の重心を変える
- なぜ自社商材なのかを、機能列挙ではなく顧客課題への対応として置く
- 情報量が多ければ slide plan を artifact で確認してから進む
- 効果と導入イメージを顧客向けに当て込み、最後に次回確認事項か判断依頼を置く

## 例3: 製造業向けで情報が揃っている入力

入力:

```text
製造業の工場向けに、工程ラベル用途で自社のラベルプリンターを提案したいです。
現場で手書きラベルが残っていて、品目記載ミスと貼り直しが課題です。
工場長と生産管理部向けの資料にしたいです。
本体、消耗品、初期設定支援を提案対象に含めます。
まずは2ライン、計4台から始めたいです。
次回訪問でそのまま使える提案資料にしてください。
```

期待する流れ:

- 追加質問は最小限またはゼロ
- 構成の重心だけ迷う場合は、summary 先行か現場課題先行かを短い選択肢で返してよい
- artifact は不要なら出さない
- 読者、営業フェーズ、課題に合わせて構成を作る
- 先頭で判断サマリーを置き、最後で次アクションを明示する
- 自社商材の押しどころが弱ければ、その確認を追加する
- 持参可能な PPTX を返す

## 例4: 情報不足で仮説前提のたたき台にする

入力:

```text
ラベルプリンターの提案資料を作りたいです。
小売向けで、店舗運用を改善したいです。
詳しい運用はまだ分かっていません。
```

返す追加質問の例:

- 値引きラベル、棚ラベル、バックヤード用途など、どの用途を想定していますか
- 誰に見せる資料ですか
- 本体だけの提案か、消耗品や支援も含めますか
- 最初にどの規模から始める想定ですか

回答が十分でない場合の扱い:

- 利用可能なら質問 UI / 質問ツールで確認を出す
- 質問 UI が使えない場合は、通常の会話で同じ確認を返す
- 用途や資料目的が複数ありえる場合は、短い選択肢で方向性を返して決めてもらう
- 一般論で断定しない
- チャットで `現時点の情報に基づくたたき台です` と明示する
- 必要なら骨格だけ artifact で見せて前提を握る
- 未確認前提を伝えたうえで資料化するか、必要なら確認事項だけ返す
- 正式提案を求められているのに不足が大きい場合は、たたき台に逃がさず確認に戻す

## 例5: 外食チェーン案件で公開情報を背景補足に使う

入力:

```text
大手外食チェーン向けに、食材ラベル用途で自社のラベルプリンターを提案したいです。
次回訪問で使える提案資料を作ってください。
```

公開情報として拾ってよい例:

- 主な業態
- 店舗展開規模
- テイクアウトやデリバリーの有無
- 公開されている食品表示やオペレーション標準化の取り組み

引き続き質問すべき例:

- 実際のラベル運用が手書きか、既存プリンターか
- 誰に見せる資料か
- 初回訪問か、正式提案か
- 本体だけか、消耗品や支援も含むか

その後の流れ:

- 公開情報から分かる背景を短く整理する
- それを案件固有の事実として断定しない
- 公開情報では埋まらない論点だけを質問 UI または通常会話で確認する
- 検索は背景補足に留め、案件条件の確定には使わない
