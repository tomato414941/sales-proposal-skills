# Examples

このファイルは、入力の強さに応じて `ready` `hypothesis-ready` `not-ready` がどう分かれるかを見るための例です。

## 例1: ready に近い案件

入力:

```text
地方物流会社向けに、現場ラベル発行の改善提案を作って。
現在は手書きラベルと旧型プリンターが混在していて、誤貼付と再印字が多い。
出荷現場の責任者と物流部長が主な読者。
3拠点に各2台ずつ導入したい想定で、まずは本体、消耗品、保守、初期設定支援を含めて提案したい。
次回訪問で使える顧客向け提案デッキにして。
```

期待する判断:

- 顧客像、課題、読者、提案対象が見えている
- `Frame` で勝ち筋を具体化しやすい
- `Challenge` を通れば `ready` にできる

期待する流れ:

- 誤貼付と再印字の業務影響を整理する
- 本体だけでなく消耗品、保守、初期設定支援を含む提案範囲を明示する
- 現場責任者向けの運用改善と、部長向けの品質・効率改善を両立する構成にする

## 例2: hypothesis-ready にすべき案件

入力:

```text
自社製品のラベルプリンターの提案資料を作って。
物流か製造のどちらかの顧客を想定している。
現場でラベル運用に課題がありそうなので、次回商談で使える資料にしたい。
```

期待する判断:

- 顧客業種、課題、提案対象がまだ抽象的
- 一般論で埋めると危険
- 仮説提案としてなら進められるので `hypothesis-ready` が妥当

期待する流れ:

- 事実と仮説をはっきり分ける
- 「物流向け仮説」か「製造向け仮説」かを決めて構成を作る
- 未確認事項を最後に明示したうえで PPTX を作る

## 例3: not-ready にすべき案件

入力:

```text
ラベルプリンターの営業資料を作って。
```

期待する判断:

- 顧客像がない
- 課題がない
- 読者がない
- 提案対象の範囲もない

この場合は:

- 完成資料に進まない
- まず確認すべき項目を返す
- `not-ready` と明示する

## 例4: RFP 対応で ready にはしない案件

入力:

```text
RFP に対する提案書を作りたい。価格はまだ未確定。
既存システムとの連携要件も確認中。
ただし次回会議で方向性は見せたい。
```

期待する判断:

- 正式提案としては早い
- ただし仮説提案としては成立する
- `hypothesis-ready` が基本

期待する流れ:

- 価格未確定と連携要件確認中を明示する
- 方向性、構成、前提条件を整理する
- 比較や効果の見せ方は強くしすぎない

## mode の見分け方

`ready`

- 顧客固有の課題が明確
- 提案対象が具体的
- そのまま持参しても誤解を招きにくい

`hypothesis-ready`

- 仮説提案としては成立
- 未確認事項を明示すれば資料化できる
- 次回確認の材料として使える

`not-ready`

- 顧客理解が弱い
- 提案対象が曖昧
- 一般論の資料しか作れない
