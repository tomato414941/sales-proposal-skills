# Proposal Rules

このファイルは、営業担当者の入力から提案資料を作るときに、何を確認し、どの条件を満たすまで次に進めないかを判断するためのルールです。

`SKILL.md` の workflow で迷ったときは、まずこのファイルを参照してください。

## 1. まず整理する骨格

最初に、入力と添付情報から次を整理してください。

- 顧客
- 提案商材
- 用途または課題
- 制約
- 想定読者
- 今回の目的

この段階では、欠けている情報があってもよいです。まず骨格を作り、何が不足しているかを見極めてください。

ここで整理するのは AI の仮説です。まだ確定事項として扱わず、`Clarify` で確認または選択肢化してください。

## 2. 必須確認項目

次が埋まるまでは `Compose` に進めません。

- 読者
- 既存運用
- 提案範囲
- 営業フェーズ

加えて、次の情報も確認対象です。

- 顧客固有の現場課題
- 自社商材の強み、押したい価値
- 導入対象規模
- 効果を語る業務または指標
- 価格、納期、連携範囲など誤解されやすい条件

## 3. 仮説可項目

次の項目は未確定でも、チャットで未確認前提を明示したうえで資料化して構いません。

- 導入対象規模
- 効果を語る業務または指標
- 一部の前提条件

ただし、これは初回訪問 / 仮説提案に限ります。正式提案、比較検討、見積・稟議前では、これらの不足を軽く扱わず確認に戻してください。

この場合は、少なくとも次をチャットで明示してください。

- `現時点の情報に基づくたたき台です`
- `未確認の前提は次回確認後に更新します`

## 4. ユーザーに確認する論点

次は、skill 側で勝手に決めずにユーザーに確認してください。

- 読者
- 既存運用
- 提案範囲
- 比較提案かどうか
- 資料の目的
- 顧客に見せる前提条件として何を明示するか
- 営業フェーズ
- なぜ自社商材を推すのか

## 5. 選択肢を返して決めてもらう論点

yes/no では決めにくく、どれを選ぶかで資料の意味や構成が変わる場合は、2〜3 個の短い選択肢を返して決めてもらってください。

典型例:

- 本部向けにするか、現場向けにするか、両方にするか
- 初回ヒアリング用か、比較提案用か、見積前提の提案にするか
- 正式提案として完成度を優先するか、初回訪問用のたたき台として進めるか
- 比較提案として組み立てるか、課題整理提案として組み立てるか
- summary 先行で見せるか、現場課題先行で見せるか
- 一度たたき台を返すか、確認事項だけ返すか

## 6. reader と営業フェーズの扱い

reader は厳密な属性スキーマにせず、少なくとも次の実務カテゴリで扱ってください。

- 本部向け
- 現場向け
- 役員向け

reader ごとに重視するものの目安:

- 本部向け: 標準化、展開性、投資判断、運用統一
- 現場向け: 作業負荷、ミス低減、使いやすさ、導入しやすさ
- 役員向け: 経営インパクト、判断論点、実行可能性

営業フェーズは少なくとも次で整理してください。

- 初回訪問 / 仮説提案
- 比較検討
- 正式提案
- 見積・稟議前

フェーズが曖昧な場合は、選択肢を返して決めてもらってください。

フェーズごとの重心:

- 初回訪問 / 仮説提案: 課題認識の共有、方向性の合意、次回確認事項
- 比較検討: 比較対象、判断論点、選定理由
- 正式提案: 提案範囲、導入条件、期待効果、実行イメージ
- 見積・稟議前: 提案範囲、条件、次の社内判断材料

## 7. Skill 側で寄せてよい論点

次は、ユーザー確認なしでも skill 側で寄せて構いません。

- スライド枚数の細かな調整
- 図表の種類
- 具体的な表現の磨き込み
- レイアウトや色の選び方

## 8. Compose 禁止条件

次に当てはまる場合は `Compose` に進めず、確認事項だけ返してください。

- 読者が不明
- 既存運用が不明
- 提案範囲が不明
- 営業フェーズが不明
- 顧客像がほぼない
- 提案対象が製品カテゴリしか分からない
- 用途や課題が一般論だけ
- 比較提案なのに比較対象が不明

加えて、資料の方向性が複数ありえるのに選択肢化していない場合も `Compose` に進めてはいけません。

## 9. Render 禁止条件

次に当てはまる場合は PPTX を生成してはいけません。

- `Compose` の必須確認項目が埋まっていない
- 顧客固有の課題が前半 3 スライド以内に反映されていない
- 提案対象が具体化されていない
- 読者に合う説得順序になっていない
- 次アクションまたは判断依頼が読み取れない
- 価格、納期、機能、対応範囲の誤読リスクが高い
- 比較提案なのに比較対象または既存運用が不明

## 10. 質問の出し方

- 利用可能なら質問 UI / 質問ツールを優先する
- 質問 UI が使えない場合は、通常の会話で同じ確認を返す
- まずまとめて質問する
- 一問ずつ思いつきで聞かない
- 提案品質に直結するものだけを聞く
- すでに入力や添付にある情報は聞き返さない

質問 UI と artifact の役割は分けてください。

- 質問 UI / 質問ツール: 必須確認項目の回収
- artifact: 骨格や slide plan の合意形成

優先順位:

- 高優先: 顧客課題、提案対象、読者、既存運用、導入前提
- 中優先: 比較対象、競合、効果測定の観点
- 低優先: デザインの好み、細かな表現

質問内容の目安:

- 課題が抽象的なら、現場の困りごとを聞く
- 提案対象が曖昧なら、本体だけか、消耗品や支援も含むかを聞く
- 自社商材の押しどころが弱いなら、競合や代替手段と比べて何を推したいかを聞く
- 読者が不明なら、誰に見せる資料かを聞く
- 効果訴求が弱いなら、既存運用や発生しているミスを聞く
- 導入イメージが弱いなら、店舗数、台数、拠点数などの規模を聞く
- 資料目的が曖昧なら、どの判断を前に進める資料かを聞く
- 営業フェーズが曖昧なら、初回訪問、比較提案、見積前提のどれかを聞く
- 正式提案か、初回訪問のたたき台かが曖昧なら聞く

## 11. 構成の品質ルール

各スライドで最低限整理するもの:

- タイトル
- 1枚1メッセージ
- 根拠
- 必要図表
- 話すポイント

ルール:

- 1枚1メッセージを守る
- 根拠が弱い場合は、チャット上で未確認であることを明示する
- 顧客固有の論点を前半に置く
- 標準章立てを機械的に繰り返さない
- 製品カタログの説明にしない
- reader と営業フェーズに合う説得順序にする
- この資料で前に進めたい判断を先頭付近で読めるようにする
- 顧客課題と提案内容の対応が読めるようにする
- なぜ自社商材なのかが読めるようにする
- 最後に次アクションまたは判断依頼を置く

複数の自然な構成案が成立する場合は、短い選択肢で確認してから確定してください。

## 12. 差し戻し条件

次に当てはまる場合は差し戻してください。

- 構成問題: `Compose`
  - 一般論のスライドが続く
  - スライド順序に理由がない
  - 製品説明だけで終わっている
  - 効果が抽象語だけで表現されている
  - 次に何を取りにいく資料か分からない
  - 自社商材を選ぶ理由が読めない
- 前提不足: `Clarify`
  - 読者、既存運用、提案範囲、営業フェーズ、比較対象、自社商材の押しどころのいずれかが足りない
  - 仮説前提が多すぎて顧客誤認リスクが高い
- 提案方向の問題: `Frame`
  - 課題と提案対象の対応が見えない
  - 読者別の説得順序になっていない
  - 営業フェーズに対して資料が重すぎる、または軽すぎる
  - 自社商材である必然が弱い

選択肢化すべき論点を勝手に決め打ちしていた場合も `Clarify` に戻してください。

## 13. 言い過ぎと unsupported claim

次の表現は、根拠が弱いまま使わないでください。

- 必ず改善する
- 完全に解決する
- 確実に実現できる
- 絶対に問題ない
- 100% 効果が出る
- リスクを排除する
- ミスをなくす

次の表現は仮説として扱ってください。

- かもしれない
- おそらく
- 可能性が高い
- 想定される
- 見込み

## 14. 生成後レビュー

PPTX 生成後に、最低限次を確認してください。ここでは見た目、誤認リスク、営業担当が説明しやすいかを中心に見てください。論理や方向性の問題は `Challenge` に戻してください。

- 汎用テンプレ資料に見えないか
- 顧客固有の論点が前半で伝わるか
- 箇条書きの羅列に逃げていないか
- 営業担当がそのまま説明できる具体性があるか
- 次アクションが自然に読めるか
- 仮説前提が残る場合、顧客に誤解を与える見せ方になっていないか

問題がある場合は `Compose` に戻してください。必要なら `Clarify` または `Frame` に戻してください。

## 15. examples と eval の分離

`examples.md` は使い方の説明用です。評価には同じ案件を使わないでください。

少なくとも次を守ってください。

- examples に載っている案件を、そのまま評価正本にしない
- 飲食、物流、製造または小売のように複数業種で確認する
- 最小入力、中程度入力、十分な入力、情報不足案件を分けて評価する
