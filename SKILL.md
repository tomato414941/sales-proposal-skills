# Proposal Design

## このスキルの役割

このスキルは、営業提案をいきなり資料化せず、まず顧客向け提案として成立するかを判定し、そのうえで提案戦略とスライド構成を設計するためのものです。

提案が成立する場合のみ、最後に組み込みの `pptx` スキルで PowerPoint を生成します。成立しない場合は、仮説提案または確認依頼として止めてください。

## 基本原則

- 事実と仮説を混ぜない
- 情報不足を一般論で埋めない
- 顧客固有性が弱い提案を完成扱いしない
- 価格、納期、機能、対応範囲を推測で断定しない
- 問題が見つかったら前段に差し戻す
- PPTX は最終派生物であり、正本は提案判断と提案構成である
- Claude Code では必要に応じて `.proposal/` に途中成果物を保存してよい
- Claude.ai では会話の見出しを使って同じ段階を維持する

## 終了 mode

このスキルは、最後に必ず次のいずれかの mode を宣言してください。

- `ready`
  顧客向け提案として成立しており、そのまま持参できる PPTX を生成してよい状態
- `hypothesis-ready`
  仮説提案としては成立するが、未確認事項と前提条件を明示して扱うべき状態
- `not-ready`
  顧客向け資料として完成扱いすべきではなく、確認事項と不足情報を返すべき状態

## Workflow

### 1. Qualify

まず、この案件を顧客向け提案として進めてよいかを判定してください。

最低限見るべきもの:

- 顧客像
- 課題または期待効果
- 提案対象
- 制約
- 想定読者

これらが弱い場合は、完成資料に進まず、確認事項を返してください。

### 2. Frame

提案の骨格を決めてください。

- 誰を説得するか
- 顧客が何を重視するか
- 何を勝ち筋にするか
- 提案対象をどこまで含めるか
- 何を言い過ぎないか

ここで顧客固有の課題、提案対象、導入前提が曖昧なら先へ進めないでください。

### 3. Compose

スライド本文の前に、構成を決めてください。

各スライドでは最低限次を整理します。

- タイトル
- 1枚1メッセージ
- 根拠
- 必要図表
- 話すポイント

ここでは枚数と順序を案件に合わせて調整し、標準章立てをそのまま繰り返さないでください。

### 4. Challenge

この段階では、提案を批判的に見直してください。単なるセルフチェックではなく、必要なら前段に戻してください。

必ず見るべき観点:

- 顧客固有性が十分あるか
- 一般論の製品紹介に崩れていないか
- 断定しすぎていないか
- 提案対象が具体化されているか
- 読者に合う説得順序になっているか
- 顧客に持っていくには危険な曖昧さが残っていないか

問題がある場合は、`Frame` または `Compose` に戻してください。

### 5. Render

`ready` または `hypothesis-ready` の場合のみ、組み込みの `pptx` スキルで PowerPoint を生成してください。

PPTX 生成時のルール:

- `Compose` で固めたスライド構成を正本として使う
- 箇条書きだけのスライドを並べない
- 案件に合う色、レイアウト、図表を選ぶ
- 役員向けなら短く、現場向けなら具体性を増やす
- 仮説提案である場合は、未確認事項を隠さない

必要なら [references/slide_design.md](references/slide_design.md) を読んでデザイン方針を決めてください。

### 6. Review

PPTX を生成したら、そのまま持参できるかを最後に確認してください。

必ず見るべき観点:

- 汎用テンプレ資料に見えないか
- 顧客固有の論点が前半で伝わるか
- 箇条書きの羅列に逃げていないか
- 営業担当がそのまま説明できる具体性があるか
- 仮説と事実の境界が崩れていないか

ここで問題があれば `Compose` に戻してください。必要なら `Frame` まで戻してください。

## Gate と差し戻し条件

次の条件を満たさない場合は、先へ進めないでください。

- 顧客固有の課題が弱い場合: `Qualify` または `Frame` に戻る
- 提案対象が曖昧な場合: `Frame` に戻る
- 各スライドの主張に根拠がない場合: `Compose` に戻る
- 一般論の製品紹介に寄っている場合: `Frame` に戻る
- 顧客に持っていくには断定が強すぎる場合: `Challenge` から差し戻す
- PPTX が汎用テンプレに見える場合: `Review` から `Compose` に戻る

`ready` にしてよいのは、顧客固有性、提案対象、説得順序、断定の強さが顧客持参に耐える場合だけです。

## Claude Code で使う場合

必要なら次のような途中成果物を `.proposal/` に残して構いません。

- `facts_and_gaps.md`
- `proposal_strategy.md`
- `slide_plan.md`
- `review_notes.md`

ただし、これらは必須ではありません。会話内で同じ段階管理ができるなら、ファイル化しなくても構いません。

## 禁止事項

- 仮説を事実として扱わない
- 価格、納期、機能、対応範囲を推測で断定しない
- 根拠の弱い主張を確認済みのように見せない
- 顧客提示不可の内容を最終資料へ入れない
- 顧客固有性が弱いまま完成扱いしない
- `not-ready` の状態で顧客向け PPTX を最終成果物として出さない
- `Review` を省略しない
- 毎回同じ章立て、同じ見た目に固定しない
