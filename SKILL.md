# Proposal Design

## このスキルがやること

このスキルは、営業提案をいきなりスライド化せず、段階的に設計してから最終的に PPTX を作るためのものです。

まず事実と仮説を分け、提案戦略とスライド構成を固め、その後でセルフチェックを行い、最後に組み込みの `pptx` スキルで PowerPoint を生成します。

## 基本ルール

- 事実と仮説を混ぜない
- 不足情報は作り話で埋めず、要確認事項として残す
- 戦略が固まる前にスライド本文を書き切らない
- セルフチェックを終えるまで PPTX を生成しない
- 顧客提示不可の内部事情は最終資料へ出さない
- Claude Code では必要に応じて `.proposal/` に途中成果物を保存してよい
- Claude.ai では会話の中で同じ段階を維持しながら進める

## 進め方

### Step 1: 事実整理

入力情報から、まず確認済み事実を整理してください。少なくとも次を確認します。

- 顧客
- 提案対象
- 課題
- 制約
- 関係者
- 期限
- 既知の根拠資料

Claude Code でファイル出力する場合は、必要に応じて `10_facts.md` 相当を `.proposal/` に保存してください。

### Step 2: 不足情報と仮説の分離

次に、不足情報、仮説、仮説の根拠、提案への影響、要確認事項を整理してください。

ここでは「分からないこと」を無理に埋めず、提案の前提として明示してください。

### Step 3: 提案戦略の策定

次を決めてください。

- 誰を説得するか
- 顧客が重視する評価軸は何か
- 何を勝ち筋にするか
- 何を言い過ぎないようにするか
- どの順番で納得してもらうか

### Step 4: 課題と解決策の対応付け

顧客課題ごとに、提案内容を対応付けてください。

- どの課題に何で応えるか
- 製品標準で対応するのか
- 個別対応が必要か
- 導入前提は何か
- 想定効果は何か

### Step 5: スライド構成の設計

スライドはまず構成から決めてください。本文の前に、各スライドについて次を整理します。

- タイトル
- 1枚1メッセージ
- 根拠
- 必要図表
- 話すポイント

ここではスライド枚数と説得順序を案件に合わせて調整してください。標準章立てをそのまま繰り返さないでください。

### Step 6: セルフチェック

PPTX を作る前に、最低限次を確認してください。

- 事実と仮説が混ざっていないか
- unsupported claim がないか
- 価格、納期、対応範囲を推測で断定していないか
- 根拠の弱い主張が明示されているか
- 顧客提示不可の内部事情が混ざっていないか
- スライド順序が対象読者に合っているか

セルフチェック結果は、Claude Code では `.proposal/60_validation.md` に保存してよく、Claude.ai では会話中に短く明示してください。

### Step 7: PPTX 生成

セルフチェック完了後、組み込みの `pptx` スキルを使って PowerPoint を生成してください。

PPTX 生成時のルール:

- Step 5 で固めたスライド構成を正本として使う
- 箇条書きだけのスライドを並べない
- 案件に合う色、レイアウト、図表を選ぶ
- 役員向けなら短く、現場向けなら具体性を増やす
- 数字があるなら数値コールアウトや比較図を使う

`pptx` スキルへ渡す前に、必要なら [references/slide_design.md](references/slide_design.md) を読んでデザイン方針を決めてください。

## Claude Code でファイル出力する場合

必要なら次のような途中成果物を `.proposal/` に残して構いません。

- `10_facts.md`
- `20_gaps_and_assumptions.md`
- `30_strategy.md`
- `40_solution_map.md`
- `50_outline.md`
- `60_validation.md`
- `70_deck_draft.md`

ただし、これらは必須ではありません。会話内で同じ段階管理ができるなら、ファイル化しなくても構いません。

## 禁止事項

- 仮説を事実として扱わない
- 価格、納期、機能、対応範囲を推測で断定しない
- 根拠の弱い主張を確認済みのように見せない
- 顧客提示不可の内容を最終資料へ入れない
- セルフチェックを飛ばして PPTX を生成しない
- 毎回同じ章立て、同じ見た目に固定しない
