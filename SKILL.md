# Proposal Design

## このスキルの役割

このスキルは、営業担当者の自然文入力を起点に、提案に必要な情報を整理し、不足している重要情報を質問で回収しながら、最終的に顧客向け提案資料まで仕上げるためのものです。

ユーザーには内部状態や mode 名を見せないでください。ユーザーが見るのは、必要な追加確認、仮説前提のたたき台、または提案資料そのものです。

## 基本原則

- 事実と仮説を混ぜない
- 情報不足を一般論で埋めない
- 足りない情報は、提案品質に直結するものだけを質問する
- スライド作成を始める前に、持参品質に必要な情報をできるだけ揃える
- 価格、納期、機能、対応範囲を推測で断定しない
- 顧客に誤解を与える資料は出さない
- PPTX は最終成果物であり、スライド構成と提案判断が先にある
- Claude Code では必要に応じて `.proposal/` に途中成果物を保存してよい
- Claude.ai では会話の見出しを使って同じ段階を維持する

## 外部 UX

このスキルは、ユーザーに対して基本的に次のいずれかを返してください。

- 資料化に必要な追加確認
- 仮説前提のたたき台
- 完成した提案資料

仮説で進める場合でも、内部 mode 名は見せず、自然文で次のように伝えてください。

- `現時点の情報に基づくたたき台として作成します`
- `未確認の前提はチャット上で明示し、資料では誤解を招かない形にします`

artifact を使う場合は、主成果物としてではなく、営業担当者と AI が提案の骨格や slide plan に合意するための補助表示として使ってください。案件が単純で確認点が少ない場合は、artifact を出さずにそのまま進めて構いません。

## Workflow

### 1. Intake

まず、ユーザー入力と添付情報から使える情報を整理してください。

最低限拾うもの:

- 顧客
- 提案商材
- 用途または課題
- 制約
- 想定読者
- 今回の目的

ここでは、欠けている情報があっても、まず骨格を作るところまで進めてください。

### 2. Frame

提案の骨格を決めてください。

- 顧客のどの課題に応えるのか
- 何を提案対象に含めるのか
- 誰を説得するのか
- 何を勝ち筋にするのか
- 何を言い過ぎないか

この時点で、持参品質に直結する不足情報を洗い出してください。

情報量が多い、説得順序を一度まとめて確認したい、またはこの方向で作ってよいかを営業担当者と早く握りたい場合は、必要に応じて骨格を artifact で見せて確認して構いません。

### 3. Clarify

スライド作成に入る前に、必要な追加確認をまとめて質問してください。

`Compose` に進む前に、少なくとも次の必須確認項目を埋めてください。

- 読者
- 既存運用
- 提案範囲

次に当てはまる場合は、`Compose` に進めてはいけません。

- 読者が不明
- 既存運用が不明
- 提案範囲が不明
- 比較提案なのに比較対象が不明

質問は、次に直結するものだけに絞ってください。

- 顧客固有の現場課題
- 提案対象の具体範囲
- 想定読者
- 既存運用
- 台数、拠点、導入前提
- 価格、納期、連携範囲など誤解されやすい条件

質問を減らしすぎないでください。特に、読者、既存運用、提案範囲、導入対象規模が抜けたままでは、一般論の提案や曖昧な導入案に流れやすくなります。

少ない入力から始めてよいですが、次が弱いままスライド作成に進めないでください。

- 誰のどの判断を前に進める資料なのか
- 既存運用で何が問題になっているのか
- 何をどこまで提案するのか
- どの規模から始める想定なのか
- 効果をどの業務や指標で語るのか

質問は一問ずつ思いつきで出さず、まずまとめて提示してください。追加質問は、回答後も致命的な不足が残る場合だけにしてください。

必須確認項目が埋まった場合は `Compose` に進んでください。

次の項目だけが未確定の場合は、チャット上で未確認前提を自然文で明示したうえで `Compose` に進んで構いません。

- 導入対象規模
- 効果を語る業務や指標
- 一部の前提条件

必須確認項目が埋まらない場合は、資料化せず確認事項だけ返してください。

### 4. Compose

必要な情報が揃ったら、スライド本文の前に構成を決めてください。

各スライドで最低限整理するもの:

- タイトル
- 1枚1メッセージ
- 根拠
- 必要図表
- 話すポイント

ここでは枚数と順序を案件に合わせて調整し、標準章立てをそのまま繰り返さないでください。

情報量が多い、スライド順や主張の順序を一度握りたい、または PPTX 前に方向確認が必要な場合は、slide plan を artifact で見せて合意を取ってから `Render` に進んで構いません。artifact は補助表示であり、主成果物ではありません。

### 5. Challenge

提案を批判的に見直してください。問題がある場合は、`Frame` または `Compose` に戻してください。

次をすべて満たすまで通過扱いにしないでください。

- 顧客固有の課題が前半 3 スライド以内に出ている
- 一般論の製品紹介スライドが連続していない
- 提案対象が具体化されている
- 読者に合う説得順序になっている
- 価格、納期、機能、対応範囲を断定しすぎていない
- 顧客に持っていくには危険な曖昧さが残っていない

構成の問題なら `Compose` に戻し、前提不足なら `Clarify` に戻し、提案方向自体が弱いなら `Frame` に戻してください。

### 6. Render

資料として成立すると判断した場合のみ、PowerPoint を生成してください。

`Render` に進める前に、少なくとも次を満たしてください。

- `Compose` の必須確認項目が埋まっている
- 顧客固有の課題が前半 3 スライド以内に反映されている
- 提案対象が具体化されている
- 読者に合う説得順序になっている
- 価格、納期、機能、対応範囲の誤読リスクが高い表現がない

これを満たさない場合は PPTX を生成しないでください。

PPTX 生成時のルール:

- `Compose` で固めたスライド構成を正本として使う
- 箇条書きだけのスライドを並べない
- 案件に合う色、レイアウト、図表を選ぶ
- 役員向けなら短く、現場向けなら具体性を増やす

仮説前提が残る場合は、チャット上で明示してください。資料本体には内部ラベルを出さず、必要なら自然な前提条件や確認事項として扱ってください。

`追加確認` を返すケースでは PPTX を生成しないでください。`仮説前提のたたき台` では、`現時点の情報に基づくたたき台です` と `未確認の前提は次回確認後に更新します` を必ずチャットで明示してください。

必要なら [references/slide_design.md](references/slide_design.md) を読んでデザイン方針を決めてください。

### 7. Review

PPTX を生成したら、そのまま持参できるかを最後に確認してください。

次をすべて満たすまで完了扱いにしないでください。

- 汎用テンプレ資料に見えない
- 顧客固有の論点が前半で伝わる
- 箇条書きの羅列に逃げていない
- 営業担当がそのまま説明できる具体性がある
- 仮説前提がある場合、顧客に誤解を与える見せ方になっていない

ここで問題があれば、構成問題なら `Compose`、前提不足なら `Clarify`、提案方向の問題なら `Frame` に戻してください。

## スライド作成開始条件

次が揃うまでは、スライド作成を始めないでください。

- 顧客像
- 顧客固有の課題または用途
- 提案対象の範囲
- 想定読者
- 既存運用
- 誤解を招く重要条件の扱い

これらが不足している場合は、先に質問してください。

## Claude Code で使う場合

必要なら次のような途中成果物を `.proposal/` に残して構いません。

- `facts_and_gaps.md`
- `proposal_strategy.md`
- `slide_plan.md`
- `review_notes.md`

ただし、これらは必須ではありません。会話内で同じ段階管理ができるなら、ファイル化しなくても構いません。

## 禁止事項

- 仮説を事実として扱わない
- 価格、納期、機能、対応範囲を推測で断定しない
- 根拠の弱い主張を確認済みのように見せない
- 顧客提示不可の内容を最終資料へ入れない
- 情報不足を一般論で埋めてスライド化しない
- 追加確認が必要なのに質問せず進めない
- 顧客に誤解を与える仮説資料を出さない
- `Review` を省略しない
- 毎回同じ章立て、同じ見た目に固定しない
