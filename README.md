# sales-proposal-skills

営業担当者の少ない入力から提案に必要な情報を整理し、不足情報を回収しながら、最終的に組み込みの PPTX 機能で顧客向け提案資料を作る skill です。提案書、提案デッキ、営業資料、RFP対応の依頼で自動起動しやすい設計ですが、確認フェーズを経ずにいきなり資料化しないことを前提にしています。

価値の中心は、AI でスライドを作ること自体ではなく、次の4点です。

- 少ない入力から前に進めること
- 顧客ごとに当て込まれた提案にすること
- 言い過ぎや誤読を防ぐこと
- 最後にそのまま持参できる PPTX まで仕上げること

## 前提条件

- Claude の skill 実行環境
- 組み込み `pptx` スキルが利用できること

## 使い方

案件情報、議事録、RFP、製品情報などを渡して、提案資料を作りたいと伝えてください。

この skill は次の順で進みます。

1. 入力から提案の骨格を整理
2. スライド作成前に、不足情報をまとめて確認
3. 必要なら中間確認として artifact で骨格や構成を確認
4. 提案戦略とスライド構成を設計
5. 内容を批判的に見直し
6. 最後に PPTX を生成
7. 生成後に持参品質を確認

主な成果物:

- 追加確認が必要な場合は、その確認事項
- 迷う論点がある場合は、選択肢付きの論点整理
- 仮説前提のたたき台
- 提案戦略
- スライド構成
- 最終 PPTX

この skill は、初回訪問で使う仮説提案と、情報がかなり揃った正式提案の両方を扱います。ただし、営業フェーズが曖昧な場合は勝手に決めず、必ず確認してから進みます。

同じ案件でも、読者や営業フェーズが違えば資料の重心を変えます。たとえば本部向けなら標準化や展開性、現場向けなら作業負荷や使いやすさを前に出します。

読者、既存運用、提案範囲、営業フェーズなどの必須確認項目が埋まっていない場合は、PPTX を作らず確認に戻ります。利用可能なら質問 UI / 質問ツールで確認し、使えない場合は通常の会話で同じ確認を返します。

初回訪問では、未確認前提を明示した仮説提案を返せます。正式提案では、不足が大きい場合はたたき台に逃がさず確認に戻します。

Claude Code では、必要に応じて `.proposal/` に途中成果物を残せますが、必須ではありません。

artifact は主成果物ではなく、営業担当者と AI が提案の骨格や slide plan に合意するための補助表示として必要な場合だけ使います。

この skill は、迷う論点を勝手に一つへ決め打ちせず、必要なら短い選択肢で整理してユーザーに決めてもらうことを前提にしています。

また、笑われにくい資料にするため、先頭で顧客課題と提案価値を示し、最後に次アクションや判断依頼が読める構成を基本にします。

examples は使い方の説明用です。評価には [evaluation_cases.md](evals/evaluation_cases.md) のような別ケース群を使い、同じ案件で過剰最適化しないようにします。
